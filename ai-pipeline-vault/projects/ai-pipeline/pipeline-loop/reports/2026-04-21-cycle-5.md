# Pipeline Cycle Report — cycle 5 (2026-04-21)

## Summary

Cycle 5 is a **mixed but meaningfully better** run. BEC-52 (force cluster rebuild) worked as intended and exposed the first real side-effect of cycle-4: the dev sector-based clusterer now produces a NEW schema (`CommunicationServices`, `Industrials`, `ConsumerDefensive-Technology-Utilities` merged) whose names **don't match BEC-38/42/53's per-cluster overrides**, so BEC-53's hidden=[64]/dropout>=0.40/mask=0.25 tightening is silently bypassed on 3-4 clusters (Energy, Industrials, ConsumerDefensive-Technology-Utilities, CommunicationServices). Despite that, the **Stage-6 backtest shows a real improvement**: moderate/sideways Sharpe 3.86 (vs 3.12 cycle-4, first positive Info Ratio in any cell at +0.93), aggressive/bull flipped from -3.81% to +0.31%, and all three profiles show higher sideways returns. Only 3 clusters were promoted, all via cold-start fallback (new cluster names have no prior champion). BEC-54 (tiebreaker ordering) never fires because 0 pairs where both candidate and champion pass the filters. Cycle 5 also introduced 7 orphan RUNNING runs from training subprocesses that BEC-55 will sweep on cycle-6 start.

## Verdict & reasoning

**Verdict: `improve`.**

Cycle 5 showed concrete, mechanism-explainable improvement in Stage 6 but ALSO surfaced concrete, high-confidence bugs that will be invisible until fixed. The improvement is robust enough to keep moving (every comparable backtest cell improved; moderate/sideways crossed zero Info Ratio for the first time; bull flipped positive for every profile), but the **blocker-stack is not cleared**:

1. **BEC-53 override names don't match BEC-52 regenerated cluster names.** `configs/default.yaml:194-257` lists `Utilities`, `ConsumerDefensive`, `CommunicationServices-Industrials`; the dev clusterer now produces `ConsumerDefensive-Technology-Utilities`, `CommunicationServices`, `Industrials`. This silently bypasses the BEC-53 regularization tightening on 4 clusters (Energy, Industrials, ConsumerDef-Tech-Util, CommunicationServices — Energy never had an override). Gap values for these are +0.09 to +0.29 (Industrials +0.29, Energy +0.26).
2. **BEC-51 adaptive threshold is cosmetic — recall still collapses at the adapted threshold.** `val_effective_threshold` drops to 0.50-0.60 but `val_recall_at_primary` stays 0.001-0.08. The min_recall=0.10 filter still rejects 6/10 candidates. We need either (a) a min_recall floor that scales with the effective threshold, (b) a recall metric computed against the top-K highest-probability samples regardless of absolute threshold, or (c) direct recall-target rescaling post-isotonic.
3. **BEC-54 tiebreaker is never exercised.** `cascading_compare` short-circuits at line 98 on every non-cold-start cluster; the iso_calibration tiebreaker at line 126+ never fires. Three options: (a) make iso_calibration win even when candidate fails filters but champion also lacks iso; (b) exclude current champion from candidate pool (so stale iso=0 champ doesn't beat fresh iso=1); (c) move iso tiebreaker above the cand_passed short-circuit.
4. **New: 2 clusters have degenerate silhouette (~0.23)** — ConsumerDefensive-Technology-Utilities (68 symbols merged from 3 sectors) and Miscellaneous (85 catch-all). BEC-49 made this visible but we haven't acted on it. Prod mode (global KMeans) would likely produce fewer degenerate clusters.
5. **Calibration T=2.5 bound hit in ~10/30 runs** — models are so overconfident that temperature scaling tops out at the upper bound and isotonic has to do all the work. Either raise the upper bound to 3.0-4.0 or reduce overconfidence in training (higher dropout, lower learning rate, longer early-stop patience).

I considered `insufficient_evidence` but rejected it: three of these findings (#1, #2, #3) are **deterministic configuration/code bugs** — they don't need more runs to confirm. The backtest improvement is real even if magnitude is noisy. `plateau` is wrong — reviewed all 7 stages, ≥3 concrete implementable changes identified.

## Suggested improvements (if verdict=improve)

1. **Stage 2 — Sync `per_cluster_overrides` with BEC-52 regenerated cluster names.** `configs/default.yaml:194-257` lists `Utilities`, `ConsumerDefensive`, `CommunicationServices-Industrials`; BEC-52's forced clusterer now emits `ConsumerDefensive-Technology-Utilities`, `CommunicationServices`, `Industrials`, `Energy`. Effect: BEC-53's `hidden=[64]`, `dropout>=0.40`, `mask=0.25`, `num_layers=1`, `max_overfit_gap=0.12` tightening is silently bypassed on 4/10 clusters — Energy gap=+0.26, Industrials gap=+0.29, ConsumerDef-Tech-Util gap=+0.10 (lucky), CommunicationServices gap=+0.13. Options: (a) add override entries for all current cluster names from `clusters.parquet` (enumerate + copy the same tightening block); (b) define a `default_override` block that applies to every cluster unless a named override exists; (c) make override matching resolve partial names (e.g. `CommunicationServices*` match both `CommunicationServices` and `CommunicationServices-Industrials`). Expected impact: the 4 bypassed clusters get hidden=64/drop>=0.40 and their gaps should drop 0.1-0.2 closer to the 0.15 target.

2. **Stage 3 — Make min_recall filter threshold-aware or use a calibration-invariant recall metric.** BEC-51 lowered `val_effective_threshold` to 0.50-0.60 (working as intended), but `val_recall_at_primary` is still computed at a fixed threshold and still collapses to 0.001-0.08 for every cycle-5 run. All 7 non-cold-start clusters were rejected as `failed_recall` or `failed_stability`. Options: (a) replace `min_recall=0.10` with `min_recall_vs_base_rate=0.5` (must capture at least 50% of the cluster's positive base rate above the effective threshold); (b) use top-K recall (e.g. precision/recall at the top 10% of predictions, invariant to calibration); (c) when BEC-51's adaptive path fires, scale `min_recall` proportionally to `(effective_threshold / nominal_primary_threshold)`. Expected impact: 5-8 more clusters pass Stage-3 filters, BEC-37/BEC-53/BEC-54 finally exercised end-to-end.

3. **Stage 3 — Move BEC-54's iso_calibration tiebreaker above the cand_passed short-circuit.** `src/evaluation/promote.py:98` returns `False, "candidate failed filters"` before any tiebreaker can fire — but if the current champion is ALSO `iso_fitted=0` and the candidate is `iso_fitted=1`, we want the calibrated candidate to win on the iso tiebreaker regardless of stability filter. Options: (a) add an explicit `if cand_iso=1 and champ_iso=0 and cand_stage in {"failed_recall","failed_stability"} and cand_stability_score > 0.2: return True, "iso candidate beats uncalibrated champion despite filter fail"`; (b) reorder `cascading_compare` so the iso check at line 126-134 runs first for the "stale champion" case; (c) exclude the current champion from the candidate pool in `_find_best_candidate` so fresh iso-1 runs win tier-1 selection. Expected impact: combined with #2, 3-5 stale iso=0 champions get replaced this cycle.

4. **Stage 1 — Add degenerate-cluster merge/split logic using BEC-49 silhouette_mean_cluster.** Two clusters now show `silhouette_mean_cluster` ~0.23: `ConsumerDefensive-Technology-Utilities` (68 symbols, merged from 3 sectors — too heterogeneous) and `Miscellaneous` (85 catch-all). Options: (a) when `silhouette_mean_cluster < 0.30`, subdivide the cluster via mini-KMeans (K=2) using the same feature set; (b) emit a warning + skip training (fallback to global model); (c) re-assign symbols to their single-nearest non-degenerate cluster centroid. Expected impact: fewer muddy training signals, cleaner per-cluster specialization; could improve test precision by 0.03-0.08 on those clusters.

5. **Stage 2 — Raise temperature-calibration upper bound from 2.5 to 4.0.** ~10 of 30 cycle-5 calibration runs hit the `T=2.5` upper bound, meaning isotonic has to do all the calibration work. The 2.5 bound was chosen to prevent pathological collapse (T>>1 flattens probs to 0.5) but `WARNING: calibration T=2.0133 produces <1% signals above 0.60, falling back to T=1.0` already guards against that. Options: (a) raise upper bound to 4.0 with the existing <1% safety check still active; (b) use per-cluster calibration bounds based on empirical probability distribution; (c) log T values hitting the bound as a counter metric in MLflow for tracking. Expected impact: better-calibrated predictions on overconfident models (4-6 clusters currently), feeding into better recall in Stage 3 filters, compounding with #2.
