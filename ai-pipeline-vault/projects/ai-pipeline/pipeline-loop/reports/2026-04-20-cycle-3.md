# Pipeline Cycle Report — cycle 3 (2026-04-20)

## Summary

Cycle 3 produced a **broken pipeline run**: BEC-42's per-cluster Optuna overrides (`hidden_size: [64, 96]`) conflict with the persisted Optuna studies in Postgres which already have trials using `hidden_size=128`. Optuna raises `CategoricalDistribution does not support dynamic value space` and `'128' not in (64, 96)`, causing **8 of 10 clusters to fail training** (BasicMaterials, CommunicationServices-Industrials, ConsumerCyclical, FinancialServices, Healthcare, Miscellaneous, RealEstate, Utilities). Only Energy and ConsumerDefensive completed training. As a result, predictions, portfolios, and the backtest all rest on **stale pre-BEC-39/BEC-42 champions** from 04-16 and 04-20T00-01:xx UTC, none of which carry `iso_fitted=1`. Stage 0 (BEC-41) and Stage 5 (BEC-40) clearly worked: only 1/190 features now has std<0.5 (vs 122/191 in cycle 2), and every profile sits at or below its sector cap. Stage 6 (BEC-43) also works: degenerate cells report `n/a (insufficient data)` and the report now exposes `Bench Ann Ret` + `Tracking Err`, though the bear-regime benchmark annualization looks wrong (+7669% — clearly a short-window annualization artifact). **This is a concrete, reproducible regression — verdict is `improve`, not `insufficient_evidence`.**

## Verdict & reasoning

**Verdict: `improve`.**

Cycle 3's most urgent problem is a concrete, fully-diagnosed bug introduced by BEC-42: the Optuna study reload crashes on 8 of 10 clusters with `CategoricalDistribution does not support dynamic value space` because historical trials in the persisted studies used `hidden_size=128` while the new search space is `[64, 96]`. This is a high-impact, reproducible regression — the entire ML training path for 80% of clusters is broken. The fix is well-scoped (either migrate study names or coerce the persisted trials). This also means BEC-37/BEC-39 (isotonic calibration + prefer-iso promotion) were not effectively re-tested for most clusters this cycle — they inherit the same blockage.

Two lesser but concrete issues: the bear-regime `Bench Ann Ret=+7669.28%` is a short-window annualization artifact surfaced by BEC-43's column addition; and the cycle-3 log contains a stuck RUNNING MLflow run (CSI at 14:53) that should be cleaned up.

I considered `insufficient_evidence` — cycle 3 is the first run exercising 7 merged PRs end-to-end and one run is normally thin evidence. But the training-failure bug is deterministic and reproducible (config + persisted study state), not a noisy ML signal. Waiting one more cycle without fixing it just produces the same failure. For the non-regressed stages (BEC-41 normalization, BEC-40 sector cap, BEC-43 metric guards), we do have clean positive evidence this cycle. So `improve` is the correct verdict.

## Cleanup recommendation

`cleanup_recommended: false`. There are 153 total MLflow runs with no recorded last cleanup. That's manageable. The `*-error` runs from 14:47 and the stuck RUNNING `1118a209…` for CSI are noise, but not yet blocking. Post-fix, once training works again, we may want a cleanup to purge the error runs — but doing it now would actually harm our ability to diagnose what failed in cycle 3. **Defer cleanup until after the training-reload fix lands**.

## Suggested improvements (if verdict=improve)

1. **Fix Optuna study reload crash caused by shrunk `hidden_size` search space (BEC-42 regression)** — Stage 2. 8/10 clusters fail every training run with `CategoricalDistribution does not support dynamic value space` because persisted studies in Postgres have trials using `hidden_size=128` while the new per-cluster override restricts to `[64, 96]`. Two concrete options: (a) bump the Optuna study name (e.g., `cluster-v2/{cluster_id}` → `cluster-v3/{cluster_id}` whenever the search space changes) so new studies start clean; (b) detect incompatible historical trials on load and delete them (or start a fresh study with a suffix like `cluster-v2/{cluster_id}-narrow` for overridden clusters). Expected impact: **training works again for all 10 clusters**, BEC-37/BEC-39/BEC-42 finally land end-to-end, and champions get updated to iso-fitted runs.

2. **Fix bear-regime benchmark annualization yielding +7669%** — Stage 6. `backtest_2026-04-20.md` shows `Bench Ann Ret = +7669.28%` in all three bear rows; this is a naive `(1+r)^(252/n_days)-1` blow-up over ~15-25-day bear windows. Either (a) clamp `bench_ann_ret` to a reasonable range (e.g., ±200%) and print `n/a (short window)` otherwise, or (b) suppress the Bench Ann Ret / Tracking Err / Info columns on rows already flagged `insufficient_sample`, or (c) annualize using the full backtest window's SPY rather than just the regime slice. Expected impact: honest, readable bear-regime rows; info ratio stops being a nonsense artifact for short regimes.

3. **Purge the stuck RUNNING MLflow run and error runs after the BEC-42 fix lands** — Stage 2/3. `cluster/CommunicationServices-Industrials` has run `1118a209...` in RUNNING state since 04-20 14:53 (and 8 `*-error` FINISHED runs at 14:47 across clusters clutter the report). Add a cleanup pass that (a) transitions orphaned RUNNING runs older than 1 hour to FAILED, (b) optionally archives `*-error` run names under a tag rather than the main experiment list. Expected impact: cleaner MLflow report, correct `mlflow_report.py` output, fewer misleading "11 runs" counts.

4. **Write per-cluster silhouette scores to `clusters.parquet`** — Stage 1. The current `silhouette_score` column is a single scalar repeated on every row. For a sector-based clustering (dev) this is fine, but for global KMeans (prod) it masks degenerate per-cluster silhouettes. Compute silhouette per sample and aggregate to per-cluster mean; store as a separate column or sidecar JSON. Expected impact: observable per-cluster cluster quality; early warning for degenerate clusters in prod mode.

5. **Audit why champion promotion kept the pre-BEC-37 runs even when newer iso-fitted runs exist** — Stage 3. Setting the BEC-42 bug aside, `trading-forecaster-ConsumerCyclical@champion` still points to run `893e3146d1c1` (04-20 00:43, no iso) even though `21b59c46` (04-20 12:25, iso=1, higher val_precision) is present. BEC-39 was supposed to prefer iso-fitted candidates, but the current champion chain ignored it. Trace the `promote.py` candidate filtering — possibly the candidate set only includes runs tagged as ensemble leaves from the latest training attempt, and since we didn't get fresh training for CC in cycle 3, the 04-20 12:xx iso-fitted runs were never offered as candidates. Expected impact: once BEC-42 is fixed, make sure the next cycle actually promotes iso-fitted runs (not just produces them).
