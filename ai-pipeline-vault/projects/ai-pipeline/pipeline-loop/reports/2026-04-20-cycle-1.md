# Pipeline Cycle Report — cycle 1 (2026-04-20)

## Summary

Pipeline completed end-to-end but surfaced three material issues: (1) only 4 of 10 cluster "champion" runs made it into the MLflow Model Registry, so predictions only cover 40% of the universe; (2) portfolio optimizer produces 100% ConsumerCyclical allocations across all three profiles because it filters/sorts by `prob_up` globally without cross-sector balance, and the per-sector constraint is toothless when every candidate shares a sector; (3) normalization step is ineffective — 121/190 selected features have post-normalization std << 0.5, meaning clipping/z-score isn't standardizing many fundamentals to unit variance. Model quality is also mediocre (val precision ~0.40, val→test precision gap 10-15 pp in several clusters).

## Stage-by-stage

### Stage 0 — Data & Features

- `features.parquet`: 2,321,613 rows, 292 cols, 504 symbols, dates 2006-05-16 → 2026-04-06. Top-20 nullable columns all have 0 nulls (ingestion looks clean on price/technical side).
- Data freshness: `data/.last_ingest` stamped 2026-04-07 with last feature date 2026-04-06. Today is 2026-04-20, so features are **14 days stale**. `make pipeline` (dev) skips ingestion by default — this is expected behavior but worth flagging.
- `features_selected.parquet`: 193 cols retained (190 features + id/symbol/date). Selection manifest lives at `data/selected_features.json`. The original feature list isn't persisted so we can't tell which filter was the biggest dropper without re-running selection.
- `features_normalized.parquet`: 193 cols. **121/190 features have std < 0.5 after normalization**, including many fundamentals (e.g. `km_returnonassets` std=0.0000, `km_incomequality` std=0.0013, `km_returnonequity` std=0.085, `obv_roc_20d` std=0.008). Means are all within |0.3| so clipping is pulling distributions in, but z-score seems to be computed on the pre-clip distribution — after clipping to [p01, p99] many features are effectively collapsed. This likely reduces gradient signal for the LSTM.

Summary table shows "promoted: 0/N" for the 6 missing clusters (except 2/9 for Energy and 1/9 for CC/HC/Misc), so the current promotion evaluation is aggressive — 6 clusters fail all tiers of the cascading filter.

### Stage 5 — Portfolio

- `data/portfolios.parquet`: 60 rows (20 moderate + 25 aggressive + 15 conservative).
- **All three portfolios are 100% ConsumerCyclical** (cluster_id/sector = ConsumerCyclical). Sector constraint of 20-30% is violated in spirit because the optimizer sorts by prob_up globally (`candidates.sort("prob_up", descending=True).head(max_positions)`) — ConsumerCyclical prob_up values are so inflated that they beat every other sector's candidates before the sector constraint can engage.
- All weights are at the **min bound (0.04 for aggressive, 0.05 moderate, 0.0667 conservative)** — i.e. equal-weight fallback. Either scipy SLSQP failed to find a feasible interior solution given the tight sector constraint (or no feasible point exists when 100% of candidates are one sector), or the returns matrix was too thin. Code path at line 205-207 uses equal-weight fallback when `returns_matrix.shape[0] < 10`.

## Verdict & reasoning

**Verdict: improve.**

Stage 3 (promotion) has a concrete silent-failure bug: 6 of 10 clusters fail to register a model despite their best run being tagged as champion, leading directly to the Stage 4/5 downstream problem where only ConsumerCyclical dominates and predictions cover 40% of the universe. Stage 5 has a separate concrete bug: the top-K-by-prob_up selection bypasses the sector constraint. Stage 0 normalization is demonstrably ineffective for fundamentals (121/190 features with std < 0.5). These are all implementable and have plausible mechanisms of improvement. Several improvements could lift the mediocre val precision in Stage 2 as well.

## Suggested improvements (if verdict=improve)

1. **Fix promotion-to-registry pipeline for all clusters** — Stage 3. Currently 6/10 clusters have a run tagged `champion: 'true'` but no entry in the MLflow Model Registry. Investigate `src/evaluation/promote.py`: check whether `mlflow.register_model()` is silently failing (e.g. clusters with no ensemble model checkpoint, registry name mismatch, or promotion filters dropping all tiers without a tier-4 fallback). Expected impact: predictions expand from 218 symbols / 4 clusters to ~500 symbols / 10 clusters, restoring sector diversity and fixing the downstream portfolio concentration.

2. **Portfolio candidate selection should be sector-balanced before prob_up ranking** — Stage 5. Change `src/portfolio/optimizer.py` lines 188-195 so candidate selection picks top-N symbols **per sector** (or per cluster) rather than a global top-K. E.g. take the top `ceil(max_positions * max_sector_weight)` per sector, then let the optimizer choose weights under the sector cap. This prevents ConsumerCyclical's inflated probabilities from dominating. Expected impact: three distinct portfolios respecting sector caps, real optimization (not equal-weight fallback), portfolio risk diversification.

3. **Fix normalization for narrow-distribution fundamentals** — Stage 0. 121/190 features end up with post-normalization std < 0.5 (many near 0), because z-score stats are likely computed on the raw distribution then clipping compresses most mass to a narrow band. Re-compute mean/std **after** percentile clipping, or switch to rank/quantile normalization for features with highly non-Gaussian distributions. Expected impact: LSTM gradient signal restored on fundamental features; model quality (val precision, stability) may lift from the ~0.40 band.

4. **Recalibrate ConsumerCyclical cluster (and audit other over-confident clusters)** — Stage 2/4. ConsumerCyclical outputs prob_up mean=0.90 with test precision only 0.39 — the model is uncalibrated. The existing temperature-calibration step is apparently not catching this. Either tighten the `signal_preservation` penalty so T can actually rise, or add an explicit monotonic isotonic/Platt calibration step using holdout predictions. Expected impact: more discriminative probabilities, fewer false-positive "high conviction" picks, portfolio stops being monopolized by one cluster.

5. **Close the val→test precision gap via stronger regularization for Utilities / RealEstate / FinancialServices** — Stage 2. These three clusters show 10-15pp val→test precision drops despite the existing overfit gap penalty. Increase `max_overfit_gap` penalty weight, reduce Optuna `hidden_size` upper bound from 128 to 96 for sector-count < 40, or raise `feature_mask_rate` to 0.15 in fixed_params. Expected impact: more robust test-time precision, more clusters clearing promotion filters without relying on the cascading fallback tiers.
