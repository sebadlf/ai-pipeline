## Verdict & reasoning

**Verdict: `improve`.** 

The dominant concrete issue this cycle is the **promotion blocker**: BEC-37's isotonic calibration pushes post-calibration probabilities to match empirical UP rates (~45-50%), so recall@primary_threshold(0.65) collapses to 0.0005-0.024 for every calibrated run. The promotion filter requires recall ≥ 0.10, so 9/10 calibrated candidates are rejected as `failed_recall` or `failed_stability`, and `cascading_compare` short-circuits at line 98 before BEC-50's iso_calibration tiebreaker can fire. Result: 9/10 champions remain pre-BEC-37 (iso=0) runs from 04-16 and 04-20 early AM, and aggregation/portfolios/backtest all run on those. Every ML-facing improvement from cycles 1-3 (BEC-37 isotonic, BEC-38/BEC-42 per-cluster overrides, BEC-50 iso tiebreaker) is effectively untested end-to-end.

Three concrete secondary issues:
1. **BEC-38/BEC-42's train-val gap target of <0.15 is not met** for 7 of 8 overridden clusters on their best ensemble member (range 0.133-0.281). The override (hidden≤96, max_overfit_gap=0.15, feature_mask_rate=0.15) is clearly too weak.
2. **BEC-49 per-cluster silhouette columns never materialized** because `clusters.parquet` pre-dates the merge (mtime 04-16) and `make pipeline` skips clustering when the file exists. BEC-49 is effectively a no-op until clustering is regenerated.
3. **Stuck `1118a209…` RUNNING run** for CSI from cycle 3 still present; cycle-4 adds ~30 FINISHED runs, MLflow total is 190.

I considered `insufficient_evidence` (first full cycle with all 10 clusters trained post-BEC-46 is thin) but rejected it: the promotion-blocker pattern is **deterministic and reproducible** (100% of post-calibration runs fail the recall filter on the 0.65 primary threshold), not ML noise. Waiting another cycle without fixing the recall/threshold mismatch will reproduce the same 1/10 promotion rate. For `plateau`: reviewed all 7 stages and found ≥3 concrete actionable items — not a plateau.

## Suggested improvements (if verdict=improve)

1. **Fix recall-filter vs isotonic-calibration mismatch blocking 9/10 promotions** — Stage 3. BEC-37's calibration reduces post-cal probabilities such that `recall@primary_threshold=0.65` drops from 0.10-0.30 (pre-cal) to 0.0005-0.024 (post-cal) — below the 0.10 filter floor. Effect: every cycle-4 run is rejected as `failed_recall`, and since `cascading_compare` at `src/evaluation/promote.py:98` returns False before the BEC-50 iso tiebreaker at line 126, no calibrated candidate can ever promote. Three options: (a) in `precision_eval.py`, compute the recall filter against the **adaptive fallback threshold** (not the fixed 0.65) when a candidate's signal rate at 0.65 is <5%; (b) lower the `min_recall_at_primary` floor from 0.10 to 0.05 post-calibration (or compute it relative to the calibrated UP-rate); (c) apply the iso_calibration tiebreaker *before* the `cand_passed != "true"` short-circuit (cascading_compare line 98) — a calibrated candidate should beat a non-calibrated champion even if its raw recall filter fails, since the uncalibrated champion's "recall" is inflated anyway. Expected impact: promotions unblock for 6-9 clusters, BEC-37/BEC-38/BEC-42/BEC-50 finally land end-to-end in aggregation.

2. **Force `clusters.parquet` regeneration so BEC-49 per-cluster silhouette columns appear** — Stage 1. `data/clusters.parquet` still has mtime 04-16 (pre-BEC-49 merge), so the new `silhouette_mean_cluster`/`silhouette_std_cluster` columns are never written, and dev reports can't see degenerate-cluster warnings. Options: (a) delete `data/clusters.parquet` on any cycle where the feature parquet mtime has advanced since the last cluster rebuild, or unconditionally on the first cycle after a schema change; (b) have `make cluster` detect a missing column and force a rebuild; (c) trigger from `Makefile` `pipeline` target by adding a schema-version file `data/.clusters_schema_version` and bumping it when columns change. Expected impact: BEC-49's observability actually becomes visible, degenerate-cluster warnings fire in prod mode.

3. **Tighten BEC-38/BEC-42 override to actually hit the <0.15 train-val gap target** — Stage 2. 7/8 overridden clusters still exceed 0.15 train-val gap on their best ensemble member (BasicMaterials 0.178, CSI 0.162, Miscellaneous 0.144, Utilities 0.181, FinancialServices 0.133, RealEstate 0.108-0.275, Healthcare 0.138-0.179). Options: (a) cap `hidden_size` to [64] alone for these clusters (currently [64, 96] — Utilities' best was h=64 with gap 0.181); (b) cap `num_layers` to 1 in the overrides; (c) raise `feature_mask_rate` from 0.15 to 0.25; (d) add a `dropout` override with floor 0.4 in the search space; (e) add a post-training reject if train_acc - val_acc > 0.20 (softer than the Optuna objective penalty). Expected impact: smaller, better-regularized models for these clusters; meaningful reduction in the val-test gap that BEC-38 was written to close.

4. **Rework BEC-50's iso_calibration tiebreaker ordering + cold-start logic** — Stage 3. Even on the 4 clusters where both candidate and champion "passed" (ConsumerCyclical, Energy, Healthcare, Miscellaneous), BEC-50 never fired because for CC/Miscellaneous the tier-1 candidate selected by `_find_best_candidate` was actually the **current champion itself** (re-selected from the iso=0 tier-1 pool). Two fixes: (a) in `_find_best_candidate`, exclude the current champion run from the candidate pool so a fresh iso=1 run wins on the iso tiebreaker; (b) move the iso_calibration check (cascading_compare lines 126-134) above the stability-score equality check, and consider moving it above the `cand_passed != "true"` short-circuit so it dominates filter failures for the specific case of "calibrated candidate vs uncalibrated champion". Expected impact: once (1) unblocks iso-calibrated candidates from the recall filter, this fix ensures BEC-50 actually decides the outcome for tie/near-tie cases.

5. **Clean up stuck `1118a209…` RUNNING run and add orphan-detection sweep** — Stage 2/3. `cluster/CommunicationServices-Industrials` run `1118a209a7f6` has been stuck in RUNNING since 04-20 17:53 (now 36+ hours old) — it inflates cluster run counts and skews `mlflow_runs_report.py`. Add a guard: on pipeline start, transition any RUNNING run older than (say) 6h to FAILED, with an informative tag `auto_cleanup=orphan`. This is the same class of cleanup BEC-48 addressed for FINISHED error runs — extend it to RUNNING orphans. Expected impact: accurate run counts, cleaner `mlflow_report.py`, fewer misleading "20 runs" entries in BasicMaterials / CSI / ConsumerDefensive where 1-2 are stuck.
